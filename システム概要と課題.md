# 地方競馬分析システム — 概要と現状の欠点

作成日: 2026-04-25

---

## 1. システム全体像

```
地方競馬場/
├── index.html              フロントエンド（ブラウザで動くすべてのUI）
├── server.py               ローカルAPIサーバー（Flask）
├── start.bat               サーバー起動バッチ
├── requirements.txt        Python依存パッケージ
├── scripts/
│   ├── fetch_races.py      データ自動収集スクリプト（GitHub Actions用）
│   ├── bot_runner.py       クラウドボット（GitHub Actions用）
│   └── backfill_conditions.py  既存JSONへの条件データ補完（一回限り）
├── data/
│   ├── YYYY-MM-DD.json     日別レース結果データ（fetch_races.pyが生成）
│   ├── bot_state.json      クラウドボットの残高・履歴
│   └── bot_settings.json   ボット設定
└── .github/workflows/
    └── bot_runner.yml      GitHub Actions スケジュール定義
```

## 2. 各ファイルの役割と内容

### `index.html`（約3,000行）

ブラウザだけで動くシングルページアプリ。Bootstrap 5 + Chart.js を使用。

| タブ | 内容 |
|------|------|
| 入力 | レース結果の手動入力・出走馬の自動取得・保存 |
| 一覧 | 保存済みレースの表示・ソート・フィルタ・条件補完 |
| 人気別 | 人気ごとの勝率・連対率・複勝率の集計と棒グラフ |
| 馬番別 | 馬番ごとの着順傾向・コース補正グラフ |
| オッズ | オッズ帯ごとの勝率分析 |
| AI予想 | 出走馬データ入力→スコアリング→推奨馬・組合せ表示 |
| 仮想ボット | ローカル自動売買ボット（ブラウザ上で動作） |

**データの保存先**: すべて`localStorage`（ブラウザ内）

**主な関数**:
- `buildHistStats()` — 過去データから人気別・馬番別統計を構築（指数減衰加重）
- `scoreHorses()` — 市場確率50% + 人気傾向30% + 馬番傾向20% で総合スコア算出
- `getRaceCondMult()` — 馬場・天候・距離・頭数・競馬場による補正係数
- `harvilleP()` / `trioHarvilleP()` — Harville公式で三連複確率を計算
- `botDecide()` — Kelly基準 or 固定額でベット判断
- `renderList()` — 一覧テーブルの描画（全15列ソート対応）
- `repairConditions()` — 欠落している馬場/距離情報をJSONから補完

---

### `server.py`（Flask API）

ブラウザからAPIで呼ばれるローカルサーバー（デフォルト: `http://127.0.0.1:5000`）。

| エンドポイント | 内容 |
|---------------|------|
| `GET /api/ping` | サーバー死活確認 |
| `GET /api/schedule` | 本日の開催競馬場一覧（keiba.go.jpをスクレイプ） |
| `GET /api/races?venue=&date=` | 指定場のレース番号一覧 |
| `GET /api/entries?venue=&date=&race=` | 出走馬（馬番・人気・オッズ）+ レース条件 |
| `GET /api/results?venue=&date=` | 全レース着順（RefundMoneyListから） |
| `GET /api/localdata/<date>` | data/YYYY-MM-DD.json をそのまま返す |

スクレイピング先: `https://www.keiba.go.jp/KeibaWeb/TodayRaceInfo/`

---

### `scripts/fetch_races.py`

GitHub Actions から1日数回実行。当日の全レースデータを`data/YYYY-MM-DD.json`に保存する。

**処理フロー**:
1. TodayRaceInfoTop → 本日の開催場リスト取得
2. RaceList → 各場のレース番号取得
3. RefundMoneyList → 全レースの着順を一括取得
4. DebaTable → 各レースの出走馬（馬番・人気・オッズ）+ 馬場・天候・距離取得
5. 既取得済みレースはスキップ（差分更新）

---

### `scripts/bot_runner.py`

GitHub Actionsから実行される仮想売買ボット。`data/bot_state.json`に結果を蓄積。

**処理フロー**:
1. 過去データ読み込み → `build_hist_stats()`で統計構築
2. 本日の開催場取得 → 各場・各レースをループ
3. 出走馬取得 → `score_horses()`でスコアリング（レース条件補正含む）
4. `bot_decide()`でベット判断（Kelly or 固定額）
5. 発走後のレースはRefundMoneyListで着順確認 → 精算
6. `data/bot_state.json`に保存 → GitHub Pagesで公開

---

### `data/bot_settings.json`（ボット設定）

```json
{
  "initialBalance": 100000,   // 初期残高（円）
  "betAmount": 1000,          // 1回のベット額
  "strategy": "ai_win",       // ai_win / kelly_win / place / trio 等
  "minOdds": 1.5,             // 最低オッズ
  "maxOdds": 20.0,            // 最高オッズ
  "venueFilter": "",          // 特定競馬場に絞る場合（空=全場）
  "useKelly": false,          // Kelly基準を使うか
  "kellyFraction": 0.25       // Kelly係数（0.25 = フルKellyの25%）
}
```

## 3. AI予想ロジックの詳細

### スコアリング式

```
総合スコア = 市場確率(50%) + 人気傾向スコア(30%) + 馬番傾向スコア(20%)
```

**① 市場確率（オーバーラウンド正規化済み）**
```
mktProb(i) = (1/odds_i) / Σ(1/odds_j)   // 25%のオーバーラウンドを除去
```

**② 人気傾向スコア**
- 過去データから人気ごとの勝率を集計（指数減衰加重 τ=90日）
- データ不足時はデフォルト値（1人気35.5%、2人気20.5%、3人気13.0%…）を使用

**③ 馬番傾向スコア**
- 過去データから馬番ごとの勝率を集計
- 帯広（ばんえい）は直線コースのため馬番バイアス除外

**④ レース条件補正**
| 条件 | 補正内容 |
|------|---------|
| 重・不良馬場 | 内枠+7%、外枠−7%、1人気−6%、6人気以下+5% |
| 稍重馬場 | 内枠+3%、外枠−3% |
| 雨・小雨 | 長距離馬+3%、短距離馬−3% |
| 短距離(≤900m) | 内枠+5%、外枠−5% |
| 長距離(>1800m) | 高人気+3% |
| 少頭数(≤7頭) | 1人気+5% |
| 多頭数(≥14頭) | 大外−5% |

**⑤ 三連複確率（Harville公式）**
```
P(A1着, B2着, C3着) = P(A) × P(B)/(1−P(A)) × P(C)/(1−P(A)−P(B))
P(三連複ABC) = Σ全順列の P(A1着,B2着,C3着)
```

## 5. 現状の欠点・課題

### 🔴 重大な問題

**① 一部競馬場のコードが未確認**
- 門別・盛岡・船橋・川崎・姫路のコードは過去90日間のスクレイピングで確認できず
- 大井(44→20)・笠松(47→23)のように他の場でも誤コードが混ざっている可能性あり
- 対策: 各場の開催日に合わせてTodayRaceInfoTopでリアルタイム確認する仕組みが必要

**② スクレイピングの脆弱性**
- keiba.go.jp のHTMLが変わるとデータ取得が全停止する
- `class='horseNum'` `class='odds_weight'` などの固定クラス名に依存している
- HTMLの変更検知・アラート機能がない

**③ データ蓄積量が少ない**
- 現在7日分（約400レース）しかなく、統計的な信頼性が低い
- 人気ごとの勝率などは最低でも1,000〜2,000レースが必要
- 特に珍しい条件（不良馬場 × 長距離など）のサンプルがほぼゼロ

### 🟡 予想精度の問題

**④ 予想がオッズに引きずられすぎ**
- 市場確率（オッズベース）の比重が50%と高く、市場の合意に収束しやすい
- 独自の「エッジ」（市場が過小評価している馬）を見つける仕組みが弱い

**⑤ 調教・血統・馬体重などの情報を使っていない**
- keiba.go.jpのDebaTableには騎手・調教師・過去走タイムなどが含まれているが未活用
- 馬体重増減（輸送疲れ等）は着順に影響するが考慮されていない

**⑥ 馬場状態の帯広への適用が不完全**
- 帯広（ばんえい）は馬場状態の取得が他場と形式が異なり、`track=null`になる
- ばんえい特有のコース（障害物・砂）に対応したロジックがない

**⑦ Harville公式の前提条件**
- 馬の独立性を仮定しているが、同厩舎・同産駒などの相関は無視
- 頭数が多い（16頭）場合、Harvilleの計算精度は理論上低下する

**⑧ 天候・馬場の変化がリアルタイムに反映されない**
- fetch_races.pyは90分ごとの実行だが、発走直前の馬場悪化は反映されないことがある

### 🟢 運用上の不便

**⑨ server.py（Flask）が起動していないと機能しない**
- 出走馬の自動取得・スケジュール確認・条件補完がすべてFlask依存
- start.bat を忘れると手動入力が必要になる

**⑩ ローカルストレージの容量制限**
- ブラウザのlocalStorageは通常5〜10MB程度が上限
- 多数のレースデータを蓄積するとパフォーマンス低下・上限超過の恐れ
- 長期運用にはデータのJSONエクスポート＆削除を定期的に行う必要がある

**⑪ 仮想ボットの精算オッズが概算**
- 単勝的中時の配当 = 記録されたオッズをそのまま使用
- 実際のレース後変動（最終オッズ）との乖離がある可能性
- 複勝・三連複の配当はヒューリスティックな推定値

**⑫ GitHub Actionsの実行タイミングと発走時間のズレ**
- Actionsは1日10回の固定スケジュール実行
- 発走10分前のオッズ（最も精度が高い）を必ずしも取得できない

**⑬ 大井・笠松のデータが手薄**
- 旧コード誤設定（大井=44→正20、笠松=47→正23）のため
- 2026年4月25日以前のデータはこの2場が欠落している

## 6. 今後の改善案（優先度順）

| 優先 | 内容 |
|------|------|
| 高 | 未確認コード（門別・盛岡・船橋・川崎・姫路）の実測確認 |
| 高 | スクレイピング失敗時のアラート機能追加 |
| 高 | データ蓄積量を増やす（過去データの遡り取得） |
| 中 | 騎手・馬体重変動を予想スコアに組み込む |
| 中 | 発走直前オッズ（15分前・5分前）を別途取得する仕組み |
| 中 | ばんえい競馬（帯広）の専用ロジック追加 |
| 低 | localStorageからIndexedDBへの移行（大容量対応） |
| 低 | 回収率の推移グラフ・的中傾向のダッシュボード強化 |
| 低 | 複勝・三連複の精算額を実払戻金で補正する仕組み |
